cendas.read_csv

cendas.read_csv(file_path: str)

Parameters/Arguments

  • file_path
    • The file path indicating where the .csv file is located

Description

Takes a file path of a .csv file as an argument. Automates the process of identifying tractcode or blockcode data within the dataset so that the codes can be read as type string as opposed to type int/float. Returns a pandas dataframe with the data from the .csv file. The cendas.read_csv method makes the 'dtype' and 'converters' arguments from the pandas.read_csv method unnecessary when it comes to making sure that the tractcodes and blockcodes are read as type string instead of type int/float.

The method assumes the following:

  • The dataframe column names are already listed out in the first line of the .csv file. This method will not work if the .csv files do not contain the column names in the first line.

Why This Method is Useful

Currently, the pandas.read_csv method reads tractcodes and block codes from US government agencies as type int by default. This causes the leading zeros to be dropped from the tractcode/blockcode data. Given the fact that the first 2 digits of every tractcode/blockcode indicate which state a tract or block is located in, dropping leading zeros will cause problems with analyzing codes starting with digits '01' all the way to '09.' For example, the tractcode '01001020100' is located in Alabama because the first 2 digits '01' correspond with the state of Alabama. However, if the leading zero was dropped because the tractcode data was read as type int or float as opposed to type string, we'd have '1001020100' as the tractcode instead, where the first 2 digits '10' correspond with the state of Delaware. The exclusion of leading zeros from tractcode/blockcode data can cause huge inaccuracies in geographic analyses.

To avoid ths, it is necessary to explcitly declare that the column names containing the tractcodes and block codes be read as strings using the 'dtype' or 'converters' arguments when using the pandas.read_csv method. This is not a terrible workaround if a pandas user is using 1 or 2 datasets. However, if a user is using many datasets, this can be very inefficient, especially given that column names containing tractcode or blockcode data can have different names across different datasets. For example, a column containing block code data in one dataset may simply be named 'block' in one .csv file and 'block_code' in a different .csv file, and the pandas.read_csv method currently requires a user to declare the exact names of all the columns they want read as a certain data types. The cendas.read_csv is a "smarter" .csv parser that automates the process of identifying the exact names of these columns and allows tractcodes/blockcodes to be read as type strings by default without the need for the user to explicitly declare this.
